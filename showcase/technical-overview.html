The fidelity contract

Five invariants, never stubbed

Every section of the architecture elaborates one of these; the lab is only useful because each is actually enforced and tested.

One token issuer

Keycloak mints every token the gateway or an MCP server validates. PingID, Athenz, Auth0, and the homegrown AS authenticate their populations; none of their tokens reach the plane directly.

PHI stays inside the boundary

The Konnect control plane sees configuration and aggregate telemetry only; vendor clouds receive only DLP-screened, allowlisted tool traffic; audit flows to the customer SIEM.

Stateless MCP from day one

No protocol session. Identity and capabilities ride every request; continuity uses explicit handles like slot_hold_id, so any request can hit any replica.

Governance at two layers

Enterprise-Managed Authorization / IdP policy governs connections — the spec scopes IdP visibility to token issuance (EMA §7.2). Kong ACLs and server scope checks govern tool calls.

Every transitional piece has an exit

The homegrown AS sunsets into Athenz + Keycloak; the vendor token broker shrinks vendor-by-vendor as ID-JAG adoption spreads; PingID's brokered leg upgrades in place to EMA.

Master diagram

Every population, token path, and flow

Solid edges are current state. Dashed labeled edges are the future EMA (ID-JAG) upgrade paths; unlabeled dashed edges feed the audit spine. The egress feed is drawn from the internal-agents node for routing clarity — developer clients reach it through the same private ingress.

Keycloak (realm mcp-plane) is the sole Resource Authorization Server, named in every server's RFC 9728 Protected Resource Metadata. Tier audiences make cross-tier replay fail cryptographically; the hub JWT stops at the component that validated it.

Spec alignment

Enterprise-Managed Authorization, mapped

The MCP Enterprise-Managed Authorization extension (stable, modelcontextprotocol/ext-auth) applies the Identity Assertion JWT Authorization Grant (ID-JAG): RFC 8693 token exchange at the IdP, then an RFC 7523 JWT authorization grant at the Resource Authorization Server. This platform was designed for that shape before it stabilized.

Claims contract

One canonical JWT, five grant paths

Kong ACLs, server permission logic, and SIEM parsing are written once against one claim vocabulary, regardless of which grant path produced the token.

Always present

iss exact issuer · aud = exactly one tier audience (mcp://tier/*) plus server URIs (mcp://srv/*) · jti joins every audit record · mcp_tier, idp_origin, mcp_contract pins · PS256/ES256 only, 5–10 minute lifetimes.

Conditional, enforced both ways

cnf sender-constraint — x5t#S256 (mTLS) on the m2m path and jkt (DPoP, RFC 9449) on the DPoP workforce path, each rejected without its matching binding · fhir_patient mandatory on the patient path, forbidden elsewhere — a hard compartment filter on every FHIR query · act carries delegation chains · directory PII is forbidden in access tokens.

Verification
