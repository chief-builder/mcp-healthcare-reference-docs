Control plane

Two Konnect hybrid control planes are created and configured through API/decK. Local Kong data planes establish mTLS uplinks through an Nginx alias proxy and cache configuration for disconnected operation.

Data plane

Requests stay on local Docker/k3d paths except real Konnect control traffic and optional real Auth0/GitHub integrations. HAPI data is synthetic. Kong first-party routes forward to container ports; egress routes forward only to registered vendor service URLs.

Application state

MCP transport sessions are disabled. FHIR calls are request-local. Scheduling continuity is a UUID handle in Postgres. Broker grants are versioned OpenBao entries, while consent transactions and single-flight locks are process-local memory.

Evidence plane

Services emit JSON audit lines. Alloy ships Docker logs to Loki. Kong exports OTel logs/traces through Collector to Loki/Tempo. Grafana queries by token ID or hub JTI.

End-to-end request sequences

Workforce clinical call

Client
Authorization code + PKCE via Keycloak, optionally brokered to fake Ping/Auth0. Receives a hub access token.
Internal Kong
Global tier pre-function requires mcp://tier/internal. The claim-driven sender-constraint plugins (cnf-check, dpop-check) pass plain bearer tokens untouched and enforce only when the token carries a binding. Rate limit applies. First-party route forwards to FHIR server.
FHIR server
A DPoP scheme shim normalizes any DPoP auth header before bearer middleware verifies signature/issuer/expiry/resource audience; when the token carries cnf.jkt, requireDpop re-validates the proof. For tools/call, required scope is checked. Tool list is group-shaped.
Authorization hook
When a patient claim exists, by-ID equality or forced search parameter is applied.
HAPI
Receives a bounded GET request. Response is capped and converted to MCP text content. Final status determines audit allow/deny.

Certificate-bound scheduling call

SPIRE helper
Writes short-lived SVID certificate/key into shared volume and rotates them.
Loop agent
Reads certificate, requests Keycloak client credentials over mTLS, validates the token’s x5t against the same certificate.
Kong :8143
Requests the client certificate. cnf-check compares leaf SHA-256 thumbprint to token claim before forwarding.
Scheduling
Revalidates token and scope, then uses Postgres for the hold. Explicit slot_hold_id carries continuity between calls/replicas.

Governed vendor call

Internal Kong
OIDC validates the egress resource audience. DLP scans the raw JSON body and records allow/deny.
vendor-token
Calls broker resolve with hub Authorization plus vendor, subject, JTI, TTL, and required scopes.
Broker
Revalidates hub JWT; reads OpenBao; returns token, consent URL, re-consent URL, or a closed failure. Refresh is single-flight per process and CAS-protected in OpenBao.
Kong
Replaces the hub Authorization header with the vendor access token.
Vendor
Receives screened MCP body and its own credential only.

OpenBao custody, single-flight refresh, DLP verdicts, and vendor-token isolation.

Upcoming MCP 2026-07-28 compliance delta

Current wire behavior

  • SDK 1.29 and 2025-era schemas.
  • Fresh stateless server/transport per POST.
  • Legacy requests without per-request identity/capability metadata are accepted.
  • No server/discover.
  • No mirrored method/name/header validation.
  • Existing test helper proves this legacy path.

Required modern behavior

  • Upgrade to the official v2 client/server packages supporting modern and legacy eras.
  • Implement server/discover and advertise supported versions/capabilities/serverInfo.
  • Require protocol version, clientInfo, and clientCapabilities in request _meta.
  • Validate MCP-Protocol-Version, Mcp-Method, and conditional Mcp-Name against the body.
  • Return modern HTTP status and JSON-RPC error codes.
  • Update all clients/tests and add a conformance gate.

Transport lifecycle

Modern HTTP exposes POST only; GET/DELETE should return 405 for older traffic. Request-scoped SSE remains optional. Client disconnect should cancel work. The FHIR fetch currently has a timeout signal but is not wired to the client disconnect signal.

Backward compatibility

If legacy support is required, v2 auto-negotiation should probe modern discovery and fall back to initialize-era behavior. The server must distinguish modern protocol errors from legacy endpoint absence.

Audit tuple and evidence path

token_id / hub_jti → principal → acting_agent → client
                     → origin_idp → tier → patient_compartment
                     → server / route → tool → decision / reason → timestamp

Server evidence

FHIR and scheduling audit the final tool outcome. FHIR distinguishes compartment, upstream exception, and upstream HTTP denial. Scope middleware audits insufficient scope.

Gateway evidence

cnf-check logs certificate-binding denials. dpop-check logs proof rejections and degraded-cache events. DLP logs both clean passes and blocks. vendor-token logs broker decisions. OpenTelemetry adds request traces.

Broker evidence

Consent, resolve path, refresh generation, stale transition, mass-stale page, revoke, security callback failures, and admin denial are JSON events.

Limitations

The lab is not an immutable SIEM, has no production retention policy, and uses unauthenticated local Loki/Tempo. Audit completeness still depends on every rejection path emitting a record.

Verification and next assessment gate

Current checks

All eight phase gates were green on the last full run (2026-07-12, 154 tests). Two TypeScript builds are clean. Production npm trees reported zero known vulnerabilities on 2026-07-10. Six read-only Phase 3 clinical/tool integration tests passed during the latest assessment. The running stack’s PRM and Origin behavior were inspected directly.

CI boundary

GitHub Actions performs offline build/compile/config parsing. npm audit is non-blocking. Live Docker/Konnect/Auth0/GitHub/k3d acceptance remains manual.

Reassessment prerequisites

  1. Modern SDK/wire migration and conformance tests.
  2. HTTPS canonical resource migration.
  3. Central contract-exact token validation.
  4. Scheduling Origin and booking-state fixes.
  5. OAuth metadata/redirect/TLS hardening.

Required evidence

Run builds, dependency audits, new unit tests, modern MCP conformance, Origin negatives, OAuth discovery/PKCE/redirect tests, phases 0–7, race tests, and a fresh token-in-log scan. Record exact versions and the deployed configuration commit.

No current certification claim: the architecture is well developed and several controls are genuinely implemented, but the current repository cannot accurately claim MCP 2026-07-28, RFC 9728, or production OAuth 2.1 compliance until the listed deltas are closed and verified.
