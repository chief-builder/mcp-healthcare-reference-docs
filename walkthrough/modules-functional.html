1. Architecture and contract documents

End-to-end reference architecture

Explains the enterprise target: one issuer, customer-controlled PHI boundary, stateless first-party servers, separate connection and tool governance, and explicit migration exits.

Its functional value is decision alignment: security, platform, identity, and compliance reviewers can evaluate the same trust boundaries and operating model.

Source: docs/mcp-e2e-reference-architecture.md

Token claims contract

Defines the JWT vocabulary used by gateways, servers, tests, and audit records: effective principal, calling client, tier, resource audiences, scopes, origin IdP, delegation, certificate binding, and optional patient compartment.

It prevents each consumer from inventing its own interpretation of identity.

Source: docs/mcp-token-claims-contract.md

Vendor token broker design

Defines how users connect SaaS accounts without placing personal access tokens on laptops or forwarding MCP hub tokens to vendors. It specifies consent, credential custody, refresh, revocation, failure behavior, and audit.

Source: docs/vendor-token-broker-design.md

Prototype plan

Turns the architecture into phases 0–7, each ending in an acceptance gate. It also names every production-to-lab substitution, including synthetic FHIR, SPIRE for workload certificates, OpenBao for vault custody, and Grafana for SIEM-like inspection.

Source: docs/prototype-plan.md

2. Identity modules

Five upstream identities normalize to one canonical JWT.

Upstream identity
Workforce, customer, workload, legacy service, or native Keycloak client.
→
Keycloak hub
Authenticates or brokers the principal and applies protocol mappers.
→
Normalized token
One issuer and one claims vocabulary, with tier and server audiences.
→
Protected plane
Gateway and MCP server independently validate the token.

realm/mcp-plane.json

The central identity configuration. It creates token scopes and mappers, normalized groups, identity-provider links, public PKCE clients, confidential service clients, and the certificate-authenticated loop-agent client.

Functionally, it ensures callers arrive at the MCP plane with consistent identity regardless of how they first authenticated. It also mints sender-constrained interactive tokens: the workforce-dpop client's tokens carry a cnf.jkt key binding, so possession of a token alone is never enough on that path.

realm/fake-ping.json

A second Keycloak realm standing in for Ping. It provides a workforce identity source whose raw group vocabulary is deliberately different, allowing tests to prove that hub normalization removes upstream-specific names.

Homegrown issuer

A deliberately small legacy client-credentials authorization server. Its tokens are not accepted by the MCP plane; Keycloak exchanges them into hub-issued tokens. This demonstrates the migration pattern for a frozen legacy issuer.

Source: compose/phase1/homegrown-issuer/

Auth0 customer identities

Patients authenticate against a real Auth0 tenant — the one leg of the lab that exercises a genuine external IdP rather than a stand-in. The hub brokers the login, and a managed user attribute links each Auth0 identity to its patient record, so customer tokens carry the fhir_patient compartment the clinical server enforces.

Source: realm/mcp-plane.json (auth0 identity provider)

Phase 1 setup

The idempotent setup script adds Keycloak permissions and user-profile behavior that cannot be represented completely by a plain realm import, including legacy token exchange and managed patient-linkage attributes.

Source: compose/phase1/setup-phase1.sh

3. Gateway and policy modules

Internal gateway state

deck/internal.yaml publishes internal FHIR and scheduling routes, an internal tier-audience wall, certificate-binding support, rate limits, and the governed SaaS egress routes.

It is the private entry path for workforce and workload clients.

External gateway state

deck/external.yaml publishes only the curated external FHIR surface. Its independent tier audience and route catalog reduce the blast radius of an internal configuration mistake.

cnf-check

When a token contains a certificate thumbprint, this plugin requires the request to arrive over TLS with the matching client certificate. A stolen certificate-bound token therefore cannot be replayed as an ordinary bearer token.

RFC 8705 property

dpop-check

When a token carries a cnf.jkt, this plugin requires a matching DPoP proof: the proof key’s thumbprint must equal the claim, and its method, URL, timestamp, and single-use identifier must hold. A sender-constrained token therefore cannot be replayed as an ordinary bearer token. Early rejection at the DP; the MCP server re-checks authoritatively.

RFC 9449 property

dlp-egress

Reads outbound request bodies before vendor delivery, applies configured patterns, blocks matches, and emits the pattern name—not the matched sensitive value—to audit logs. A body that cannot be scanned fails closed.

MRN and SSN lab patterns

vendor-token

Calls the broker with the validated hub JWT, receives the user’s vendor credential, and replaces the upstream Authorization header. The vendor sees its own token rather than the MCP-plane token.

token isolation

Plugin schemas

Each plugin directory includes a Kong schema describing allowed configuration. Setup scripts register those schemas with the internal Konnect control plane before declarative gateway state is synchronized.

4. First-party MCP server modules

Patient-compartment FHIR reads and stateless scheduling holds.

FHIR clinical server

Exposes five curated read tools over a HAPI FHIR R4 backend: patient read, patient-everything, and searches for observations, conditions, and medication requests.

It performs OAuth bearer validation, filters tool visibility by group, challenges for per-tool scopes, forces customer searches into the token’s patient compartment, caps result count and response size, and audits final outcomes.

Source: servers/fhir-clinical/

Scheduling server

Exposes find, hold, confirm, and release tools. A hold returns an explicit slot_hold_id; hold state lives in Postgres so either replica can continue the workflow.

The module demonstrates application-level continuity without an MCP session. It is a synthetic scheduling model, not a production booking engine.

Source: servers/scheduling/

Shared OAuth resource-server logic

Both servers publish Protected Resource Metadata and verify JWT signature, issuer, expiration, algorithm, and their configured resource audience using the authorization server’s JWKS.

The two source copies are currently identical rather than shared as a package.

Audit emitters

Both servers write one JSON record for each tool decision, including token ID, principal, acting agent, calling client, origin IdP, tier, patient compartment, tool, decision, and timestamp.

5. Vendor token broker modules

Vault custody, single-flight refresh, DLP verdicts, and vendor-token isolation.

Resolve
Gateway asks for a vendor credential for the validated hub subject.
→
Consent if absent
Browser authorization uses state and PKCE.
→
Vault custody
Access and refresh tokens are stored per vendor and hub subject.
→
Refresh safely
One in-process refresh wins; Vault CAS prevents stale writes.
→
Vendor call
Gateway injects only the vendor token after DLP passes.

broker/app/main.py

Owns the HTTP API and state machine: resolve, consent start, OAuth callback, grant listing, grant revocation, admin vendor inspection, proactive refresh, stale-grant detection, and audit.

broker/app/hub.py

Independently validates the incoming hub JWT before the broker trusts its subject. This prevents the gateway from becoming the broker’s only authentication boundary.

broker/app/vendors.py

Loads the reviewed vendor registry and implements authorization-server discovery, code exchange, refresh, vendor user lookup, and revocation. GitHub-specific behavior is isolated here.

broker/app/vault_store.py

Provides the OpenBao KV v2 storage adapter, including compare-and-swap writes, metadata deletion, subject listing, and separate read access for vendor client credentials.

broker/registry.json

The allowlist of enabled vendors, OAuth endpoints or metadata locations, scope ceilings, user lookup, and revocation strategy. It makes vendor onboarding a reviewed configuration change.

Mock vendor

A deterministic vendor authorization server and MCP endpoint. It issues 60-second access tokens, rotates refresh tokens, burns a family on replay, enforces PKCE, supports RFC 9207 issuer identification and RFC 7009 revocation, and exposes test counters.

Source: compose/phase5/mock-vendor/

6. Runtime, workload, and observability modules

Compose phases 0–5

Each phase is a cumulative runnable environment. Phase 0 starts identity, FHIR, Postgres, and observability. Later phases add brokered identity, Kong tiers, MCP servers, certificate-bound workload identity, and governed egress.

Provisioning scripts

The setup scripts create or reuse Konnect control planes, provision DP certificates, synchronize decK state, generate lab PKI, register plugin schemas, provision OpenBao, and deploy SPIRE plus the loop agent.

Loop agent

Runs continuously in k3d. On each iteration it rereads the current SPIFFE certificate, gets a certificate-bound Keycloak token, checks the token thumbprint, then finds, holds, and releases a scheduling slot over mTLS.

It proves secretless workload identity and certificate rotation without a pod restart.

Synthetic clinical data

seed-synthea.sh downloads a pinned Synthea build, generates synthetic patients, and posts transaction bundles to HAPI. The committed output is test data, not real PHI.

Audit collection

Grafana Alloy tails phase-5 container logs into Loki. Kong exports traces and plugin logs through OpenTelemetry Collector. Tempo stores traces and Grafana provisions the data sources and audit-tuple dashboard.

Container images

The Node servers use multi-stage builds and run as the non-root node user. The broker and loop agent use slim Python images and run as UID 65534. Server images include PRM-based health checks.

7. Verification modules

Phase 0–2

Verify baseline token claims and FHIR data, identity normalization across grant paths, tier replay rejection, gateway resilience during control-plane uplink loss, and decK drift.

Phase 3

Verifies role-shaped tool lists, scope step-up, patient-compartment enforcement, and cross-replica scheduling continuity using explicit hold handles.

Phase 4

Verifies zero secrets in the workload manifest, correct certificate-bound token claims, rejection without or with the wrong certificate, successful matching-certificate calls, and live certificate rotation.

Phase 5–7

Verifies consent, DLP, single-flight refresh, revocation, fail-closed vault behavior, end-to-end audit joins, replay and scope attacks, RFC 9207 defenses, stale-grant alerts, log-secret absence, and broker non-issuance.

GitHub Actions

Performs offline TypeScript builds, Python compilation/import checks, JSON/YAML parsing, and non-blocking production dependency audits. Live integration gates remain manual because they require Docker and external systems.

Current verification caveat

The test MCP client uses the legacy wire shape. It verifies present application behavior but does not verify the 2026-07-28 modern protocol headers, metadata envelope, discovery RPC, or version errors.

known assessment finding

Verified present state

Present: single Keycloak issuer for the plane, two gateway tiers, independent server JWT validation, audience checks, group/tool restrictions, step-up 403s, patient filter, Postgres hold handles, mTLS-bound workload path, DLP, vendor credential isolation, Vault CAS, and audit pipeline.
Lab substitutions: HTTP endpoints, development Keycloak/OpenBao modes, SPIRE stand-in for Athenz, synthetic FHIR, mock vendor, local rate limiting, and no FIPS enforcement.
Not yet compliant: MCP 2026-07-28 wire behavior, RFC 9728 canonical HTTPS resource identifiers, full claims-contract validation in the MCP servers, and Origin enforcement in scheduling.
